## 今日の目的

今日は、生徒AIの設計を教育シミュレーションへ組み込める形で固める。

主な論点:

- 生徒AIの責任範囲
- 認知モデル、個人特徴モデル、LLM発話生成器の分担
- 教師AIへ内部状態を直接見せない設計
- 一次方程式に限定しない汎用化の可能性
- 論文で主張できる範囲

## 作成したファイル

```text
docs/student_ai_design.md
```

この仕様書では、生徒AIを次の3層で定義した。

```text
生徒AI = 認知モデル + 個人特徴モデル + LLM発話生成器
```

## 設計判断

### 教育シミュレーションに入れてよいか

入れてよい。

ただし、論文や発表では「人間学習者の完全な代替」とは言わない。

安全な表現:

```text
一次方程式学習に限定して、理解度・誤概念・個人特徴を制御できる代理生徒として用いる。
```

### 汎用化できるか

不可能ではない。

ただし、全教科・全単元を一つの認知モデルで扱うのは現実的ではない。

現実的な設計は次。

```text
共通基盤 + 単元別プラグイン
```

共通化できる部分:

- 生徒ID管理
- 学習履歴
- 自己効力感
- 質問傾向
- モチベーション
- Big Five
- LLM発話生成
- ログ保存
- 観察情報の要約
- 教師AIへの情報受け渡し

単元ごとに差し替える部分:

- スキル定義
- 問題データ
- 採点器
- 誤概念マップ
- 正答確率モデル
- 誤答生成ロジック
- 学習更新ルール
